# WorkBuddy 获客 Skill 设计材料

本目录不是一个可直接安装的 Skill。它是后续由 AI 或开发者生成 WorkBuddy「云计算获客 Skill」时必须读取的需求与接口材料。

## 文件清单

| 文件 | 用途 |
| --- | --- |
| `01-云计算获客调研设计.md` | 固定的业务目标、调研流程、评分原则和边界。 |
| `02-CRM导入接口说明.md` | CRM 实际字段、导入边界和字段映射。 |
| `schemas/cloud-lead-list.v1.schema.json` | WorkBuddy 交付的机器可读 JSON Schema。 |
| `03-策略卡模板.md` | V1 的云计算调研策略卡；可替换产品、场景、信号和评分。 |
| `04-输出样例.md` | 仅用于理解结构的虚构样例；不得作为真实客户资料。 |
| `05-获客清单MD输出样例.md` | 面向销售阅读的 Markdown 成品样例；与 JSON 样例使用同一组虚构数据。 |

## 后续生成 Skill 时的硬约束

1. 把 `03-策略卡模板.md` 作为可替换资产；不要把产品场景、信号或评分权重硬编码到提示词。
2. V1 **不做客户规模、所有制、行业或客户类型的自动剔除**。它只记录公开可得的企业属性，并按云场景相关性、近期变化和证据质量排序。
3. 所有外部事实必须关联证据；“云需求”“痛点”“推荐方案”必须标记为推测，不能写成客户已确认的事实。
4. 评分必须按策略卡的固定权重计算，最好由脚本完成；模型不得自行调整权重。
5. V1 仅调用只读的外部信息工具。团队客户占用、全局客户排除和目标客户范围策略，留待后续接入内部 MCP 后实现。
6. V1 只生成 `stage: "lead"` 的候选客户；不得生成或导入销售跟进记录、客户关系、真实上云现状、账单结构、成交概率或已确认痛点。
7. 交付前必须校验 `cloud-lead-list.v1.schema.json`；CRM 侧仍须展示导入预览并由销售确认。

## 推荐交付包

```text
cloud-lead-list.v1.json      # 必须：通过 Schema 校验，供 CRM 导入
evidence.json                # 可选：如证据量大，可将证据拆分为独立文件
lead-list.md                 # 必须：给销售快速阅读
lead-list.html               # 可选：用于筛选、汇报或分享
```

若使用独立 `evidence.json`，主 JSON 的 `evidence_file` 应填写该文件名，且所有 `evidence_ids` 必须可在该文件中解析。
