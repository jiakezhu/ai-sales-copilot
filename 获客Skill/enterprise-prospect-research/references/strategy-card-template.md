# 产品与获客策略卡

每次运行先填写。未指定项采用默认值并在报告披露，不要写死到 Skill。

## 1. 任务边界

| 项目 | 内容 |
| --- | --- |
| 任务名称 | 例如：数据库弹性与云原生潜客研究 |
| 产品/能力 | 必填；产品名称和能力边界 |
| 目标数量 | 默认 20 家 |
| 候选池目标 | `max(目标数量 × 5, 60)`，上限默认 300 |
| 地区/行业 | 用户未限定时保持开放 |
| 强/普通信号窗口 | 默认 90 / 180 天 |
| 候选来源 | 指定名单、企业库、公开市场或混合 |
| 硬排除条件 | 仅用户明确要求、主体不可靠或法律/合规限制 |

## 2. 产品假设

| 项目 | 内容 |
| --- | --- |
| 解决的问题 | 不写功能清单，写客户经营或技术问题 |
| 可观察使用场景 | 公开信息中能够识别的工作负载或变化 |
| 目标购买角色 | 业务决策者、技术决策者、使用/影响角色 |
| 典型替代方案 | 自建、其他厂商、人工流程或维持现状 |
| 首轮可验证问题 | 不预设答案的发现式问题 |

## 3. ICP 与降权指标

### 正向 ICP

记录可从公开信息观察的业务模式、产品形态、技术负载、地域扩张、团队能力和采购特征。每项注明“为什么与产品相关”。

### 降权指标

降权不等于排除，例如：信号过期、业务规模与场景不匹配、招聘岗位为外包、采购已结束、集团证据无法归属到当前主体。

### 反向 ICP

只记录有证据的明显不匹配。除非用户明确要求，不得仅因规模、所有制、行业或集团属性自动排除。

## 4. 场景与信号词典

为每个产品场景填写：

| 场景 | 强信号 | 中信号 | 弱信号 | 负向/替代解释 | 目标角色 |
| --- | --- | --- | --- | --- | --- |
| 示例：数据库弹性 | 近期数据库扩容/迁移招采 | 数据团队扩张、产品流量增长 | 数字化业务 | 岗位为维护存量系统、项目已中标结束 | CTO、架构负责人、DBA |

强信号必须是近期、具体、可定位的事件；“属于互联网行业”不能定义为强信号。

## 5. 五通道配额

配额用于防止单一来源垄断，不是硬性比例：

| 通道 | 默认候选池占比 |
| --- | ---: |
| ICP 圈选 | 35% |
| 事件驱动 | 25% |
| 招聘驱动 | 15% |
| 采购驱动 | 15% |
| 相似与生态扩展 | 10% |

某通道不可用时将配额分配给其他通道，并在报告说明。

## 6. 数据源和运行预算

记录可用 MCP、网页来源、每个工具分页上限、重试次数和不可用时的替代来源。默认同一查询最多重试 2 次，发现阶段最多 3 轮。

## 7. 历史反馈（可选）

只使用聚合指标：联系率、会议率、有效商机率、主要淘汰原因、不同信号的转化表现。不得把客户沟通原文、预算或私有关系写入输出。

## 8. 完成标准

- 候选池达到目标或明确说明停止原因；
- A/B 客户存在可定位、未过期的事件证据；
- 同一事件不重复计分；
- 每家客户都有未知项和首轮确认问题；
- JSON、Markdown 和质量审计结果一致。
