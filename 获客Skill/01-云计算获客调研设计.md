# 云计算获客调研：V1 设计

## 1. 目标

通过公开、可追溯的信息，为销售生成一份**具有云计算相关信号的企业候选清单**。系统要回答的不是“某公司一定会购买云”，而是：

1. 它的业务或技术特征可能对应哪一种云计算使用场景；
2. 它近期是否出现了可验证的业务或技术变化信号；
3. 这些判断分别由哪些公开证据支持；
4. 销售第一轮应确认什么，而不是替销售虚构结论。

V1 交付的是 `Lead`（候选线索），不是已确认的 `Opportunity`（商机）。

## 2. V1 范围与非目标

### 范围

- 面向公开可检索到的企业候选；V1 不按企业规模、所有制、行业或客户类型自动剔除。
- 面向云计算相关场景：AI 算力、云原生、弹性流量、数据平台、出海部署、安全与容灾、云成本优化。
- 使用企业信息、官网、公开新闻、公开招聘和其他合法授权的数据源。
- 输出候选客户、证据、评分、需求假设和销售发现问题。

### 不做

- 不访问、发送或修改 CRM 数据。
- 不生成真实销售跟进、联系人建联状态、预算、采购周期、成交概率或客户已确认的痛点。
- 不将公开线索直接认定为客户的真实云用量、现有云厂商或账单结构。
- 不接入团队级“客户占用/撞客排除”MCP；该功能以及目标客户范围规则留给后续版本。

## 3. 核心方法：先核验主体，后研究信号

```text
策略卡加载
  → 候选企业发现
  → 企业主体与公开属性核验
  → 公开信号与证据收集
  → 云场景假设
  → 固定规则评分与分层
  → JSON / Markdown / HTML 交付
  → CRM 人工预览、确认后导入
```

V1 的企业属性（地区、行业、规模、所有制、集团关系等）是调研信息，不是自动排除条件。后续接入内部 MCP 后，才在这一流程中增加“客户占用校验、团队排除名单和目标客户范围”步骤。

### 3.1 主体与信息完整性核验

对每家候选企业，尽可能记录：

- 标准企业名称、统一社会信用代码、官网域名和总部所在地；
- 公开行业、主营业务、产品、规模、融资、集团关系等；
- 信息来源、发布时间、抓取时间和简短摘录。

信息不足时使用 `manual_review`，含义是“公开证据不足以可靠判断”，不是“不属于目标客户”。

### 3.2 云场景识别

不要用“该公司需要云”这种笼统结论。应优先识别下列一种或多种具体场景：

| 云场景 | 公开可观察线索 | 首轮销售需要确认的问题 |
| --- | --- | --- |
| AI 算力 / MLOps | 招聘算法、推理、MLOps、GPU、数据工程岗位；发布 AI 功能 | 模型训练/推理是在本地、私有云还是公有云？峰值与成本压力如何？ |
| 云原生 / 容器化 | 招聘 DevOps、SRE、Kubernetes、平台工程师；技术博客 | 是否有容器平台、运维人力或稳定性方面的改造计划？ |
| 弹性流量 / CDN | 电商大促、游戏开服、直播、内容平台、营销活动 | 高峰流量、延迟、带宽成本和容灾压力是否突出？ |
| 数据平台 | 招聘数据工程、实时计算、数据分析；数据产品上线 | 数据规模、实时性、合规与分析成本是否存在压力？ |
| 出海部署 | 海外业务、新海外站点、跨境合作、多语言产品 | 目标国家、时延、网络、安全与本地合规要求是什么？ |
| 安全 / 容灾 | 安全岗位、等保公告、业务连续性声明 | 当前备份、异地容灾和安全治理是否存在缺口？ |
| 云成本优化 | 融资收紧、降本公告、扩张后效率诉求 | 是否存在资源闲置、成本异常或账单优化目标？ |

以上仅是销售发现假设。只有招聘、公告、官网或合法数据源中明确出现的内容才能当作事实。

## 4. 信号与评分

### 4.1 信号强度

- **强信号**：近 90 天招聘云原生 / MLOps / GPU 等岗位；公开招投标或采购提及云、算力、迁移、容灾；新 AI 产品或明确海外扩张。
- **中信号**：近 180 天融资后扩张、技术团队扩招、重要产品上线、业务量或数据规模增长。
- **弱信号**：仅有软件、互联网或数字化服务等业务特征。

同一事实来自多个来源只能计一次分；交叉来源可提高证据质量，但不能重复叠加商机分。

### 4.2 首版评分模型

总分为 100 分，由 `03-策略卡模板.md` 中的权重计算：

| 维度 | 上限 | 含义 |
| --- | ---: | --- |
| 业务与云负载相关性 | 40 | 公开业务和产品特征与云计算场景的相关程度。 |
| 技术场景相关性 | 25 | AI、云原生、数据、流量、出海等具体技术场景的证据。 |
| 购买时机信号 | 25 | 近期招聘、产品、扩张、采购或变化事件。 |
| 证据质量 | 10 | 来源可信度、时效性和证据完整度。 |

分层规则：

- **A**：总分至少 70，且至少 1 条近 90 天强信号。
- **B**：总分至少 50，且至少 2 条独立中信号。
- **C**：已有一定业务关联，但时机信号不足；适合后续培育。
- **manual_review**：主体或公开证据不足，不能可靠排序。

## 5. 数据源与工具边界

V1 可只使用以下能力：

1. 一个主企业信息 MCP：企业主体核验、行业/规模/风险/集团等公开属性。
2. 公开网页、新闻或官网检索：发现业务变化和产品信息。
3. 公开招聘检索：发现技术和增长信号。

选择一家企业信息服务作为主来源，其他服务只做交叉核验或补字段。工具返回应尽量保留：

- 企业名称、统一社会信用代码、官网域名；
- 原始链接或来源引用；
- 发布时间与抓取时间；
- 与结论直接相关的简短摘录；
- 数据源名称与置信等级。

任何工具都应只读。不要让模型自行发邮件、建立 CRM 客户、修改客户阶段或写入销售记录。

## 6. 交付标准

每个候选客户至少需要：

1. 企业身份与公开属性；
2. 匹配的云场景；
3. 可追溯的公开事实和信号；
4. 明确标为“推测”的需求假设；
5. 固定规则得出的得分、等级和扣分原因；
6. 销售首轮可验证的问题；
7. 未知或冲突的信息。

以下内容必须留空或只在“待确认问题”中出现：真实上云现状、云账单、客户关系、实际预算、销售阶段、销售推进记录、已确认痛点。

## 7. 首轮验收

先选一个云场景，而不是一次覆盖所有云产品。例如：

```text
调研场景：AI 算力 / 云原生
候选来源：企业信息库、官网、公开招聘和公开新闻
样本：20–50 家
```

销售逐条评估“该公司是否值得进一步调研、判断错在事实还是信号”，再调整策略卡。首轮验收建议记录：

- 公开事实是否准确、有来源；
- 销售认可为值得进一步联系的比例；
- A/B/C 分层是否符合人工判断；
- 每类错误信号及需要降低/提高的权重。
