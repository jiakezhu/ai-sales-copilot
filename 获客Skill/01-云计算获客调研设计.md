# 企业潜客研究 v2 设计

## 1. 目标

v2 不只回答“哪些企业符合条件”，而是回答：

1. 哪些企业与指定产品存在可观察的业务或技术场景；
2. 哪些近期事件说明现在值得进一步研究；
3. 证据是否属于正确主体、是否过期、是否只是转载；
4. 哪些结论是事实、推测或未知；
5. 销售第一轮应该验证什么。

交付仍是公开研究 `lead`，不是已确认商机。

## 2. 核心升级

### 结构化产品策略

每次任务建立产品价值、ICP、反向 ICP、购买角色、强/中/弱/负向信号和时间窗。行业、规模、所有制和集团属性默认只用于理解和排序，除非用户明确设为硬条件。

### 五通道发现

候选不再只来自一次企业库筛选，而是并行使用：ICP 圈选、事件驱动、招聘驱动、采购驱动、相似与生态扩展。默认先召回 `max(目标数量 × 5, 60)` 家，最多三轮扩展，再收敛到目标数量。

### 主体和集团归属

先核验标准名称、信用代码、官网、注册地和集团关系。母子公司可以分别研究，但证据必须归属实际主体；无法解决的主体或事实冲突进入 `manual_review`。

### 证据和时效

同一底层事件只计一次。原始公告、官网和监管/招采原文优先于聚合转载。A/B 客户的评级依据必须落入带 `sourceUrl` 的新闻、招聘、招采或资质记录。

### 反向审查

完成初评后再主动寻找：同名企业错配、集团证据借用、事件过期、岗位外包、项目已结束、供应商/采购方角色混淆和与产品无关的替代解释。发现问题必须降级或转人工复核。

## 3. 评分

总分 100：产品匹配 30、场景证据 25、购买时机 25、证据质量 15、行动可验证性 5。

- A：至少 75 分，且有 90 天内可定位强信号；
- B：至少 55 分，且有两条独立中/强信号；
- C：主体可靠并有产品关联，但时机不足；
- manual_review：主体、证据或冲突不足，不进入 JSON。

评分不是成交概率，不得生成预算、采购周期或赢单概率。

## 4. 工具策略

企信慧眼优先发现，天眼查和企查查交叉核验，官网、新闻、招聘和招采原文补充事件。工具只读；失败最多重试两次，再切换来源。不可用、脱敏、分页截断和时间缺失必须在报告披露，不得用模型知识补齐。

## 5. 输出与边界

唯一 JSON 格式仍为 `crm-customer-list.v1`，直接使用 CRM 原生 `customers[]`。公开事实进入现有情报字段；需求和痛点只能作为明确推测；销售私有数组保持为空。

同一研究结果生成 `lead-list.md`，增加五通道覆盖、工具盲区、评分扣分、反向审查和停止原因。

## 6. 质量验收

交付前执行：

```text
node scripts/validate-crm-json.mjs <json>
node scripts/audit-prospect-quality.mjs <json>
```

前者递归执行正式 Schema 的必填、类型、枚举、长度、数组和额外字段约束；后者检查重复主体、摘要数量、A/B 证据时效、未知项、首轮问题和推测标识。

## 7. 后续反馈

用户可提供聚合的联系率、会议率、有效商机率和淘汰原因，用于调整关键词、通道配额和评分解释。私有沟通记录、预算和关系不得回填到公开潜客 JSON。
