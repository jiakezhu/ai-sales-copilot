# 企业客户深度调研

研究一个已指定企业主体，不做批量获客。只使用公开、合法、可定位的信息；不得写 CRM、外联、绕过登录、收集非公开个人信息或把推测写成事实。

## 1. 确定任务和三份输出

从上下文提取企业名称、产品/销售场景、地区、时间窗和报告语言。仅当同名主体无法区分且无法通过工具解决时提问。

固定生成且只生成以下三份主文件：

```text
company-deep-research.json     CRM 原生单客户事实源；文件名可自定义
company-deep-research.md       完整可审阅报告，含证据清单
company-deep-research.html     独立、可打印的精美报告，含证据清单
```

JSON 文件名不参与格式判断；内部 `schema_version` 必须为 `crm-customer-list.v1`。Markdown 和 HTML 必须由 `scripts/render-research.mjs` 从同一 JSON 生成，不得分别手写造成事实漂移。

## 2. 锁定企业主体

先读取 `references/research-playbook.md` 的“主体锁定”。至少用两个独立来源确认标准名称，并尽量确认统一社会信用代码、注册地、法定代表人和官网域名。

主体未锁定前不得研究股权、人员、新闻或风险。同名、集团母子公司、品牌与运营主体冲突时，建立冲突记录；无法消解则停止交付，不得拼接不同主体的数据。

## 3. 规划来源和 MCP

读取 `references/mcp-orchestration.md`，先检查当前可用工具，再按以下顺序工作：

1. **企信慧眼**：`qxb_op_enterprise_resolve` 锁定主体；专项信息先查 `qxb_data_catalog_get`，再用 `qxb_op_api_spec_get` 获取真实参数，最后调用 `qxb_op_api_call`。
2. **天眼查**：`search_companies`、`get_company_basic_profile` 锁定主体；专项信息必须先调用 `get_company_capabilities`，只使用其返回的真实工具名。
3. **企查查**：交叉核验身份、股东、人员、变更、风险等关键事实；仅在名称和信用代码同时具备时调用 `verify_company_accuracy`。
4. **原始公开来源**：企业官网、监管/交易所、法院/政府、招采原文、招聘原页和主流新闻，用于补足可点击证据。

不得猜测 MCP 工具名、目录 ID、API 名或参数。工具不可用时继续使用剩余来源，并在 `source_coverage` 中记录 `inaccessible` 和影响。

## 4. 建立研究矩阵

读取 `references/research-playbook.md`，逐项研究并填写覆盖状态：

- 工商身份与历史变更；
- 股权穿透、最终受益人、实际控制人、质押/冻结；
- 母子公司、对外投资、分支机构和集团边界；
- 董监高、核心人员、任职轨迹和公开人员关系；
- 产品、业务模式、客户/渠道、合作伙伴、竞争与供应链；
- 技术栈、网站/应用、专利、软著、商标和资质；
- 招聘岗位、地点、职能、技术词和时间趋势；
- 新闻、融资、产品、扩张、合作、管理层变化时间线；
- 招投标中的采购方/供应商角色、金额和项目状态；
- 司法、行政、经营、税务、舆情与合规风险；
- 销售相关假设、反证、未知项和首轮确认问题。

每个维度必须是 `verified`、`partial`、`not_found`、`inaccessible` 或 `not_applicable` 之一。空数组不能代替覆盖说明。

## 5. 维护证据和主张

读取 `references/evidence-standard.md`。先建证据账本，再写结论：

- 证据 ID 使用 `E001` 递增；主张 ID 使用 `C001` 递增；
- 每条重要主张必须引用证据 ID；推测还必须写反证、替代解释或待确认项；
- MCP 结果没有公开 URL 时，必须记录工具名、查询条件、记录 ID 和可复查定位信息；
- 同一底层事件的转载只保留一个事件，其他来源只用于交叉核验；
- 来源冲突不得静默合并，写入 `conflicts[]`；
- 不引用搜索结果摘要替代原文；摘录保持简短，不复制长篇受版权保护内容。

来源优先级：官方登记/监管/司法/招采原文 > 企业官网与正式公告 > 企业信息库 > 主流媒体 > 聚合转载。

## 6. 做结构穿透和时序分析

股权穿透至少追踪到自然人、国资/上市公司、基金最终管理实体或无法继续穿透的明确原因；保留每条路径、直接/间接持股比例、截至日期和证据。遇到循环持股、比例缺失或多层基金，不得推算不存在的最终比例。

人员关系仅记录公开任职、共同投资、法定代表人、董监高、历史任职和公开业务联系。不得推断私人关系、亲属关系或未公开联系方式。

默认观察窗：

- 工商/股权：当前快照 + 近 24 个月重要变更；
- 新闻/招采/风险：近 24 个月，重大历史事项可回溯；
- 招聘：近 12 个月，按职位去重并分析趋势；
- 销售信号：近 180 天优先。

## 7. 从强制脚手架填写 JSON

读取 `references/output-contract.md`。**禁止从空白文件手写 JSON，禁止自行发明字段名，禁止复制旧版结构。** 先运行 Schema 驱动的脚手架生成器：

```text
node scripts/create-research-scaffold.mjs --company "法定企业名称" --query "用户原始查询" --out company-deep-research.json
```

只能在生成文件的既有字段中填值或复制数组项。所有 `__FILL_*` 必须替换；没有可靠信息的可选字段、对象或数组模板项必须整项删除。不得把占位符、空壳事实或猜测当作研究结果。不得删除 `deepResearch` 的必填键，也不得把 `events` 改名为 `news`、把 `intellectual_property` 改名为 `ip`。

顶层必须是 `crm-customer-list.v1` 且仅有一个客户。公开研究识别到的 `orgChain[]` 人物统一使用 `relationStatus: "identified"`。事实、主张、图谱边和事件必须引用 `evidence_ids`；销售研判只能包含明确标记的假设、反证、未知项和发现式问题。CRM 私有销售字段与跟进数组保持为空。

## 8. 使用唯一交付入口

研究完成后只运行：

```text
node scripts/finalize-research.mjs company-deep-research.json --out .
```

该命令依次执行占位符检查、CRM JSON Schema 校验、研究质量审计、临时目录渲染和三文件一致性验证。任一步失败都会停止，且不会用错误结果覆盖目标 Markdown/HTML。修正源 JSON 后重新运行同一命令。

不得手写 Markdown 或 HTML，不得绕过失败命令，不得在没有看到“交付门禁通过”和“交付完成”时宣称任务完成。`render-research.mjs` 与 `verify-deliverables.mjs` 自身也会重新执行门禁。若运行环境没有 Node.js 或无法执行脚本，只能报告交付被阻止，不能降级为自由生成三份文件。

## 9. 交付门槛

交付前确认：

- JSON 文件名可自定义，但内部必须是 `crm-customer-list.v1` 单客户包且可由 CRM 直接预览导入；
- 三份文件存在且公司名称、研究 ID、主张和证据数量一致；
- Markdown 与 HTML 均包含完整证据清单、冲突和数据盲区；
- 股权、人员、招聘、新闻四个核心维度均有结论或明确缺失原因；
- 高置信主张至少有一个高权威来源或两个独立来源；
- HTML 为单文件、无外部脚本依赖、响应式、可打印、链接安全；
- JSON Schema、质量审计和三文件一致性检查全部通过。

如核心主体仍冲突、证据引用断裂、三份文件不一致或 HTML 未通过检查，不得宣称完成。
