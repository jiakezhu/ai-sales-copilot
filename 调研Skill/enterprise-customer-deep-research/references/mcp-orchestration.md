# MCP 编排规范

## 通用原则

先枚举当前会话真实可用的工具，再调用。不得根据产品名称猜工具名、目录 ID、API 名或参数。所有调用均为只读；记录查询条件、调用时间、结果记录 ID和失败原因。

## 企信慧眼（QXB）

1. 用 `qxb_op_enterprise_resolve` 解析并锁定主体。
2. 专项数据先调用 `qxb_data_catalog_get` 找到真实目录。
3. 用 `qxb_op_api_spec_get` 读取接口参数。
4. 最后用 `qxb_op_api_call` 请求股权、人员、变更、风险、知识产权等数据。

不得跳过目录和接口规范直接猜参数。无公开 URL 时，证据 `locator` 至少写明工具、查询主体、目录/API 和记录 ID。

## 天眼查（TYC）

1. 用 `search_companies` 找候选主体。
2. 用 `get_company_basic_profile` 核对基础资料。
3. 调用 `get_company_capabilities` 获取该主体当前可用能力。
4. 只调用能力返回中真实存在的专项工具。

## 企查查（QCC）

用于交叉核验名称、信用代码、状态、股东、人员、变更和风险。只有法定名称与统一社会信用代码均已获得时才调用 `verify_company_accuracy`。其他专项调用也必须以会话中真实暴露的工具为准。

## 公开来源

优先打开原始页面：国家企业信用信息公示系统、交易所/监管、法院/政府、招采平台、企业官网/公告、招聘原页和主流媒体。搜索结果摘要只用于发现线索，不作为最终证据。

## 故障降级

- 单个 MCP 不可用：使用另外两个企业库和原始公开来源交叉核验。
- 登录墙/验证码/权限不足：不得绕过；将覆盖状态记为 `inaccessible`。
- 三个企业库均不可用：仍可做公开来源研究，但身份与股权结论不得标为高置信，除非有权威原始来源。
- 调用返回空：记录查询条件与 `not_found`，不要将“未检索到”写成“不存在”。
